Angular Complete Info 02

💡 왜 이벤트바인딩이 필요할까? 컴포넌트의 모든 프로퍼티는 내부에서만 액세스할 수 있기 때문에 외부에서는 액세스가 불가능하다. 그렇기 때문에 부모 컴포넌트가 프로퍼티에 바인딩할 수 있도록 하려면, 해당 엘리먼츠 프로퍼티에 decorator @Input을 추가해야한다.

  • Overview
    • HTML Elements: Native Properties & Events
      • 이벤트 바인딩 프로퍼티는 HTML 엘리먼츠 및 네이티브 프로퍼티 및 이벤트에서 사용가능하다.
    • Directives: Custom Properties & Events
      • 이벤트 바인딩 프로퍼티는 Directive에서도 사용 가능하고, ngClass, ngStyle에서도 사용 가능하다.(use property binding)
    • Components: Custom Properties & Events
      - 이벤트 바인딩 프로퍼티는 컴포넌트에서 사용하고, 자체 커스텀 프로퍼티 및 커스텀 이벤트에 바인딩할 수 있다.
      -
      ###1. Property & Event Binding

  1. @Input → Custom Properties 이 컴포넌트를 사용하여 외부로부터 부모 컴포넌트로부터 프로퍼티를 바인딩 할 수 있다

      // parent component html file
      <app-server-element
        *ngFor="let serverElement of serverElements"
        [srcElement]="serverElement"
      ></app-server-element>
    

      // child component ts file
      @Input() srcElement: {type: string, name: string, content: string};
  • @Input('srcElement') element;: 부모 컴포넌트에서 주입된 변수가 자식 컴포넌트에서는 타당하지 않거나 변경하고 싶을 때 사용할 수 있다.
    • 이제 자식컴포넌트에서는, 부모 컴포넌트에서 주입된 값을 element 프로퍼티로 사용할 수 있다. srcElement는 더이상 작동하지 않는다.

  1. @Output → Binding Custom Event 이 컴포넌트를 사용하는 부모 컴포넌트가 새로운 eventEmitter로 생성한 자신의 이벤트를 수신할 수 있도록 한다.

        // parent component ts file
        onServerAdded(serverData: {serverName: string, serverContent: string}) {
          this.serverElements.push({
            type: 'server',
            name: serverData.serverName,
            content: serverData.serverContent
          });
        }
    

        // parent component html file
        <app-cokpit
          (serverCreated)="onServerAdded($event)"
        ></app-cokpit>
    

        // child component ts file
        @Output() serverCreated = new EventEmitter<{serverName: string, serverContent: string}>;
    

        onAddServer(){
          this.serverCreated.emit({serverName: this.newServerName, serverContent: this.newServerContent});
        }

  • eventEmitter의 특징

    • eventEmitter는 TypeScript의 generic type.
    • eventEmitter는 Angular 프레임워크 내의 객체로, 이벤트를 내보낼(애미터 할) 수 있다.

  • @Output(’serverCreated’) svCreated = new EventEmitter();: 부모 컴포넌트에서 주입된 메서드 이름이 자식 컴포넌트에서는 타당하지 않거나 변경하고 싶을 때 사용할 수 있다.

    • 이제 자식컴포넌트에서는, 부모 컴포넌트에서 주입된 값을 svCreated 프로퍼티로 사용할 수 있다. serverCreated는 더이상 작동하지 않는다.
  1. ISSUE
  • 서로 같은 레벨에 있는 두 개의 컴포넌트(A,C) 간에 커뮤니케이션을 하려면, A 컴포넌트에서 이벤트를 내보내고, B 컴포넌트(부모 컴포넌트)에서 무언가를 변경한 다음, 새로운 데이터를 C 컴포넌트로 전달해야한다 → (입력과 출력의 체인의)복잡성 증가

2. View Encapsulation의 이해

  • CSS의 속성이 각 컴포넌트에 묶이는 현상으로 이는 Angular에 의해 강제된 동작이지 브라우저의 기본 동작이 아니다.

  • Angular가 View Encapsulation을 적용하는 방법

    • 브라우저에서 CSS 속성을 확인하면, 각 DOM에 _ng-content-ejo-2 같은 속성이 추가되어 있는 것을 확인할 수 있다. 이는 Angular에 의해 적용된 것이다.
    • Angular가 View Encapsulation을 하는 방법은, 단순히 컴포넌트의 모든 프로퍼티에 동일한 속성을 부여하는 것이다. (같은 클래스를 가진 DOM의 경우, 같은 프로퍼티가 추가됨을 확인할 수 있다)
    • style selector를 변경하면 자동으로 attribute selector가 컴포넌트에 대해 정의된 모든 스타일에 추가하여 이러한 스타일이 해당 컴포넌트의 엘리먼츠에만 적용되도록 할 수 있다.

  • View Encapsulation은 일종의 shadow DOM을 모방한다. shadow DOM은 모든 브라우저에서 지원이 되지 않기 때문에 Angular가 이를 모방하여 뷰를 캡슐화한다.

  • 만약 View Encapsulation을 사용하고 싶지 않다면?

      @Component({
        ...
        encapsulation: ViewEncapsulation.None
      })
    • ViewEncapsulation.Native: shadow DOM 기술
      • ViewEncapsulation.Native 속성을 사용한다면, shadow DOM이 에뮬레이트된 이전과 같은 결과를 얻을 수 있다. 그러나 에뮬레이트된 브라우저만을 서포트하고 있다는 것을 명심해야 한다.
    • ViewEncapsulation.Emulated: default

3. 템플릿에서 Local Reference의 사용

  <input
    type="text"
    class="form-control"
    **#serverNameInput**
  />
  <button (click)="onAddServer(**serverNameInput**)"></button>
  • local reference는 HTML 엘리먼트에 배치할 수 있다. 따라서 input element 뿐만 아니라 template에 표시되는 모든 엘리먼트에 해시태그와 함께 추가한 후, 이름을 추가한다. 이러한 참조는 이 엘리먼트에 대한 참조이다. (위의 #serverNameInput 의 경우, input 엘리먼트의 참조이다)
    • 이 참조는 엘리먼트에 대한 참조이기 때문에, 입력된 값만 포함되는 것이 아니라 HTML 요소 전체에 대한 모든 속성을 포함한다.
  • Local Reference의 경우, 템플릿 내의 모든 장소에서 사용할 수 있으나, TypeScript 코드 내에서는 사용할 수 없다.
  • Local Reference: 템플릿 내의 일부 엘리먼트에 액세스하여 템플릿 내에서 직접 사용할 수 있는 매우 편리한 기능이다. 또한 여기서 Local Reference된 엘리먼트의 값을 출력할 수도 있고, TypeScript에서 사용하는 경우와 마찬가지로 메서드를 이용하여 이 값을 전달할 수도 있다.

  • 예외케이스

    • Local Reference를 이용하여 메서드의 인자 안에 Local Reference를 넣어 DOM을 직접적으로 가지고 올 수도 있으나, 우리가 메서드를 사용하기 전에 DOM을 가지고 와야할 케이스가 있다. (메서드 호출 전 DOM 접근 방법)
    // html
    <input
      type="text"
      class="form-control"
      #serverContenstInput
     />
    

     // typescript
     @ViewChild('serverContenstInput') serverContenstInput: elementRef;
    

     onAddServer(nameInput: HTMLInputElement) {
       this.serverCreated.emit({
         serverName: nameInput.value,
         serverContent: this.serverCOntentsInput.nativeElement.value,
       });
     }

  • ViewChild 데코레이터를 사용하여 메서드 호출 전 DOM에 접근이 가능하다.

    • ViewChild는 인자가 필요하다 → 위의 코드에는 serverContenstInput (Local Reference)를 넣어주었다.
    • 타입은 elementRef이다. → Local Reference와 달리 템플릿에서 메서드를 이용하여 전달하던 엘리먼트 자체를 직접적으로 전달한다.
    • serverContenstInput 프로퍼티는 네이티브 엘리먼트에 직접적으로 액세스하여 input element를 가지고 올 수 있다. → ViewChild 템플릿의 DOM 엘리먼트에 직접 액세스 가능
    • 한가지 주의할 점은, 인자를 변경해서는 안된다. → this.serverContentsInput.nativeElement.value = 'Something' // You can't do that
    • 그러나 ViewChild 처럼 DOM에 직접적으로 액세스하는 방법을 좋지 않다. Directive를 이용해라. → DOM에서 출력하고 엘리먼트를 직접 조작하지 않으려면, string Interpolation 및 property binding을 사용하는 방법도 있다.

4. ng-content → 컴포넌트 안에서 콘텐츠 투영 방법

```tsx
// child component html
<div class="panel panel-default">
    <div class="panel-heading">{{ element.name }}</div>
    <div class="panel-body">
        <ng-content></ng-content>
    </div>
</div>


// parent component html
<childComponent>
    **<div>123123</div>**
</childComponent>
```
  • ng-content: directive로, selector와 이 컴포넌트를 훅으로 사용하여 컴포넌트 내부에 배치해 Angular에게 위치를 표시할 수 있다.
  • 사용처
    • 각 탭에 다른 소스에서 가져온 콘텐츠가 존재하고, 항상 property binding을 통과하고 싶지 않은 탭 위젯과 같이 재사용이 가능한 위젯을 구축하는 경우 좋은 기능으로 사용될 수 있다.
    • Angular가 XSS를 방지하기 위해 HTML 태그를 이스케이프 처리하고 그 문제를 해결하기 때문에, 복잡한 HTML 코드인 경우, property binding을 사용하는 것보다 ng-content를 사용하여 표시하는 것이 더 좋은 방법이다.
      • 왜? property binding을 사용하는경우, 프로퍼티 안에 스크립트가 들어갈 수 있어 직접 방지를 해줘야하지만, ng-content를 사용하는 경우, Angular가 XSS 방지를 도와주기 때문에 더 최적의 방법이 될 수 있는 것.

5. Component Lifecycle

  • What is Component Lifecycle?

    • Angular에서 새로운 컴포넌트가 생성되고, Angular가 예를 들어 selector 중 하나를 발견하였을 때, 이러한 컴포넌트를 생성하는 경우 해당 컴포넌트의 새 버전을 인스턴스화하여 DOM에 추가한다. 따라서 새로운 컴포넌트가 인스턴스화되면, Angular는 프로세스 몇가지를 거쳐 코드를 실행할 수 있게 된다.

  • Component Lifecycle Method

    1. ngOnChange → 바인딩된 @input 프로퍼티가 변경된 후에 호출된다.
    • 새로운 컴포넌트가 생성되었을 때 바로 실행되지만, 그 후 바인딩된 @input 프로퍼티 중 하나가 변경될 때마다 항상 호출되는 메서드이다. (@input이 새로운 값을 받을 때)
    1. ngOnInit → 컴포넌트가 초기화될 때 호출된다. (contructor → ng OnInit call)
    • 컴포넌트가 초기화 될 때 실행된다. 이 뜻이 우리가 렌더링된 화면을 볼 수 있다는 것을 의미하는 것은 아니다(아직 DOM에 추기되지 않았다)
    • Angular는 기본적인 초기화가 완료되어 우리의 프로퍼티에 액세스하여 초기화할 수 있다는 것이다.
    1. ngDoCheck → change detection이 될 때마다 호출된다.
    • ngDoCheck도 여러번 호출된다. 이 메서드는 change detection이 실행될 때마다 실행되기 때문에 실제로 많이 호출된다.
    • 개발모드로 실행될 때는 2번 실행되는 것처럼 보인다. (change detection 사이클이 1개 더 존재하기 때문에)
    • ngDoCheck는 자주 실행되지만, 강력한 코드를 실행하는 것이 아니라면, change detection가 자주 실행된다는 것은 문제가 되지 않는다.
    • change detection 란?
      • 단순히 Angular가 컴포넌트 템플릿 또는 컴포넌트 내부에서 어떤 것이 변경되었는지 판단하기 위한 시스템으로, 어떤 프로퍼티 값이 1에서 2로 변경되어 템플릿에 다시 출력되는 부분, 다시 렌더링 하는 부분이 존재할 때마다 ngDoCheck는 호출된다.
      • 주의할 점: 변경사항이 있는 경우 뿐만 아니라, 변경사항이 없는 버튼을 클릭하여도 ngDoCheck가 실행하게된다.
      • 왜? Angular는 변경사항이 있는 지 확인해야 하기 때문에 어디선가 클릭하거나 타이머가 켜지거나 observale이 되는 것과 같은 특정 트리거 이벤트를 체크해야한다. 이런 경우 또한 코드를 체크하고 ngDoCheck가 실행된다.
      • 매우 비효율적일 수도 있으나, Angular는 매우 효율적인 방식으로 이 작업을 수행하므로 change detection 기능이 매우 뛰어나고, 큰 비용이 들지 않는다. ngDoCheck는 모든 change detection 사이클에 대해 Angular에게 수동으로 통지하는 등의 작업을 수행하려는 경우 매우 유용한 방법이다. 수동으로 하지 않을경우, Angular가 탐지하지 못한 변경이 존재할 수 있다.
    1. ngAfterContentInit → ng-content가 view에 투영된 후 호출된다.
    • ngAfterContentInit은 ng-content를 통해 투영된 콘텐츠가 초기화될 때마다 호출된다. 따라서 컴포넌트 자체의 뷰가 아니라 상위 컴포넌트의 뷰, 특히 ng-content를 통해 컴포넌트에 추가되는 부분을 뜻한다.
    1. ngAfterContentChecked → ng-content가 변경될 때마다 호출된다.
    • ngAfterContentCheck는 change detection 시 컴포넌트에 투영하는 이 콘텐츠가 변경될 때마다 실행된다.
    1. ngAfterViewInit → 컴포넌트 및 자식 뷰가 초기화되면 호출된다.
    • ngAfterViewInit은 자체 컴포넌트의 뷰가 초기화가 완료되고 뷰가 렌더링되면 호출된다.
    1. ngAfterViewChecked → 뷰(및 자식 뷰)가 변경될 때마다 호출된다.
    • ngAfterViewChecked는 뷰가 변경될 때마다 호출된다. 따라서 해야할 모든 변경이 뷰에 표시되었는 지, 또는 Angular에 의해 검출되지 않았는지 확인할 수 있다.
    1. ngOnDestroy → 컴포넌트가 삭제되려고 하면 호출된다.
    • 컴포넌트를 파괴하는 경우, 예를 들어 컴포넌트에 ngIf를 배치하고 이것이 false로 설정되어 DOM에서 제거되면 ngOnDestroy가 호출된다. 이 메서드는 정리 작업 바로 전에 호출되기 때문에 정리 작업을 수행할 수 있다.

6. Component LifeCycle Example

  // parent component
  export class AppComponent {
    onChangeFirst() {
        this.serverElements[0].name = 'Changed!';
    }
  }


  // parent component html
  <app-server-element
    [name]="serverElement.name">
  </app-server-element>


  // child component
  export class ServerElementComponent implements OnInit, OnChanges {
    @Input() name: string;
  }
  • app-server-element로 serverElement 객체를 전체로 보내면, serverElementComponent 으로 변경된 값이 바인딩된 프로퍼티로 전달되지 않는다.
  • serverElements 내부의 객체의 값을 변경하더라도, serverElementComponent의 name 값은 컴포넌트에서는 갱신되어야 하지만,실행해보면 ngOnChangs는 트리거되지 않는다. → 왜냐하면, 내부의 값이 변경되었기 때문에 둘 다 메모리 내에서 같은 장소를 가리키고 있기 때문이다.
  • 즉, @input에 바인드하고 있는 프로퍼티는 변경되지 않았으며, 값 또한 변경되지 않았다. (ngOnChanges 가 트리거되지 않았기 때문에)
  • 그렇기 때문에 우리는, reference type인 serverElement를 보내는 것이 아닌, primitive type인 serverElement.name을 바인딩함으로써 ngOnChanges도 트리거되고, 프로퍼티의 변경값도 적용시킬 수 있다.
  //server-element html file


  <div>
    <div #heading></div>
  </div>


  //server-element ts file
  @ViewChild('heading') header: ElementRef;


  ngOnInit() {
    console.log(this.header.nativeElement.textContent) // empty
  }


  ngAFterViewInit() {
    console.log(this.header.nativeElement.textContent) //TestServer
  }
  • ngAFterViewInit에 도달하기 전에는 Local Reference인 heading을 사용할 수 없게된다.
  • ngOnInit을 사용하면 템플릿 엘리먼트에 엑세스를 하여 값을 사용할 수 있지만, 아직 렌더링이 되지 않았기 때문에 DOM에 있는 일부 엘리먼트의 값을 확인할 수 없다.

7. @ViewChild()

  @ViewChild('serverContentInput') serverContentInput: ElementRef;


  // use
  @ViewChild('serverContentInput', {static: true}) serverContentInput: ElementRef;
  • static 인자의 뜻
    • {static: true}: ngOnInit에서 선택한 엘리먼트를 액세스할 계획이 있다
    • {static: false}: ngOnInit에서 선택한 앨리먼트에 액세스하지 않을 예정이다.

8. @ContentChild()

  <p #contentParagraph></p>


  @ContentChild('contentParagraph') paragraph: ElementRef
  • ContentChild 데코레이터는, ng-content 내부의 값을 부모컴포넌트가 아닌(부모컴포넌트에서 주입되므로) 자식컴포넌트에서 액세스하고 싶을 때 사용한다. 그러나 이는 View의 일부가 아니므로, ngAfterViewInit 라이프사이클에서는 작동하지 않는다.
  • selector를 전달할 수 없으므로 contentParagraph이 여기에서 선택한 셀렉터가 될 것이고, viewChild와 마찬가지고 elementRef 유형이 될 일부 속성에서 이것을 입력할 수 있다. contentInit에 도달하기 전에는 값이나 그 어떤 것에 접근할 수 없다.
