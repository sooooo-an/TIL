4. Service & Dependency Injection

  1. 왜 Service가 필요할까?

  • 서비스는 코드 복제 및 데이터 저장, 데이터를 제공하기 위해 사용한다. 또한, 서비스를 이용하면, 코드를 중앙 집중화할 수 있다. 그 뿐만 아니라, 서비스를 활용하여 데이터 저장소를 만들어서 관리하는 사용자 서비스를 만들 수 있다.

  • 변경된 구조

  • 서비스가 필요한 이유

    • 컴포넌트간 커뮤니케이션: UserDetailComponent의 경우, AboutComponent의 데이터를 전달받아야하는데, 이 과정을 진행하기 위해서는 AboutComponent 데이터가 필요없는, AppComponent에게 이벤트를 바인딩하여 UserComponent로 데이터를 보내야한다.
  1. Logging Service 생성하기 → Using TypeScript
    export class LoggingService {
        logStatusChange(status: string) {
            console.log('A server status changed, new statue ' + status);
        }
    }


    // new-account
    import { LoggingService } from '../logging.service';


    export class NewAccountComponent {
        coCreateAccount() {
            //....
            const service = new LoggingService();
            service.logStatusChange(accountStatus);
        }
    }
  • 위의 코드는 TypeScript를 이용하여 만든 서비스로 이는 Angular에서 서비스를 사용하는 방법이 아니다. 일반적으로 Angular는 서비스에 액세스하는 훨씬 더 나은 방법을 제공(dependency injector)하므로 인스턴스를 수동으로 생성해서는 안된다.
  1. 컴포넌트 안에 Logging Service 주입하기 → Using Angular

  • Angular의 Dependency Injector(Hierarchical Injector) 이용하기

    • Dependency는 클래스가 의존하는 것이다. 예를 들자면, NewAccountComponent는 loggingService에 의존하고 있다. 왜냐하면, 우리는 해당 서비스에서 메서드를 호출하고, Dependency Injector는 단순히 이 클래스의 인스턴스를 우리의 컴포넌트에 자동으로 Dependency을 주입시켜준다. 우리가 해야할 일은, 그러한 인스턴스가 필요하다는 것은 Angular에게 알리는 것 뿐이다.

  • 그렇다면, 이러한 인스턴스가 필요하다는 것을 Angular에게 어떻게 알릴까?

    • constructor에 클래스를 추가하고, 우리가 사용하려는 서비스를 컴포넌트에 추가시키면 된다.

  • Angular Service Example

        export class LoggingService {
          logStatusChange(status: string) {
            console.log('A server status changed, new statue ' + status);
          }
        }
    

        // new-account
        import { LoggingService } from '../logging.service';
    

        export class NewAccountComponent {
          constructor(
            private loggingService: LoggingService
          ) {
          }
          coCreateAccount() {
            //....
            const service = this.loggingService;
            service.logStatusChange(accountStatus);
          }
        }

  • constructor에 클래스를 작성하여, Angular에게 LoggingService가 필요함을 알릴 수 있다.

  • Angular에 알리는 것을 왜 constructor에서 해야할까?

    • 누군가 우리에게 NewAccountComponent의 인스턴스를 제공한다고 생각해보자. 이 NewAccountComponent 결국 TypeScript 클래스이므로, 앱 어딘가에서 무언가가 발생하도록 인스턴스화를 해야하기 때문에.

  • 그럼 누가 컴포넌트를 만드는 책임을 가지고 있을까?

    • Angular가 가지고 있다. Angular는 템플릿에 selector를 배치하고, Angular가 이런 selector를 만나면, 컴포넌트의 인스턴스를 제공한다. Angular는 컴포넌트를 인스턴스화하는 책임이 있으므로, 컴포넌트를 올바르게 구성해야한다.
    • 즉, constructor에게 인자를 정의하면, Angular가 이를 인식하고, 해당 인자를 제공하려고 한다. 따라서 타입을 정의했기 때문에, loggingService의 인스턴스가 필요하다는 것을 알 수 있다. 그렇기 때문에 타입핑을 꼭 해주어야 한다.

  • 이제 Angular는 우리가 무엇을 원하는지 안다. 그러나 인스턴스를 제공하는 방법을 모른다.

  • Angular가 인스턴스를 제공하는 방법

      @Component({
        selector: 'app-new-acount',
        templateUrl: '',
        styleUrl: '',
        providers: [LoggingService],
      })

  • Angular는 컴포넌트를 분석할 때, LoggingService를 제공할 수 있어야함을 인식하고, 수행할 수 있도록 스스로 설정한다.

  • 그 후, 실제로 컴포넌트를 빌드하고 구축하면, 이러한 인스턴스가 원하는 것을 알 수 있으며, 이러한 인스턴스를 제공하는 방법을 알 수 있으므로 컴포넌트 내에서 간단하게 실행할 수 있다.

  • 인스턴스를 수동으로 작성하는 것이 아니라 Angular가 직접 작성하는 것이 더 나은 이유는 무엇일까?

    • 기본적으로 Angular는 에코시스템에 머물고, Angular는 앱의 동작을 알고 있기 때문에
  1. Data Service 생성하기
    // account.service.ts
    export class AccountsService {
        accounts = [
            {
                name: 'Master Account',
                status: 'active'
            },
            {
                name: 'Testaccount',
                status: 'inactive'
            },
            {
                name: 'Hidden Account',
                status: 'unknown'
            },
        ];


        addAcount(name: string, status: string) {
            this.accounts.push({name: name, status: status})
        }


        updateStatus(id: number, status: string) {
            this.accounts[id].status = status;
        }
    }

  • 더이상 AppComponent를 통해서 코드를 전달할 필요가 없다. AppComponent Template에 있는 account 데이터를 전부 삭제해도 된다.

    • AppComponent Template
       // app.component.ts
       @Component({
          ///....,
          provider: [AccountsServcie],
        })
        export class AppComponent {
         accounts: {name: string, status: string}[] = [];
    

         constructor(private accountsServcie: AccountsServcie) {}
    

         ngOnInit() {
           this.accounts = this.accountService.accounts;
         }
        }

  • provider 배열에 AccountsService를 추가하여 Angular가 이런 AccountService를 생성하는 방법을 알 수 있도록 해야한다.

  • 서비스에 있는 accounts는 배열이다. 이는 참조타입이므로, 실제로 서비스에 저장되어 있는 accounts와 동일한 배열에 액세스된다.

  • newAccountComponent TypeScript Example

         // new-account.component.ts
      export class NewAccountComponent {
        onCreateAccount(accountName: string, accountStatus: string) {
          this.accountService.accAcount(accountName, accoutStatus);
        }
      }

  • accAcount 이벤트가 바인딩되면, 오류가 발생하지 않고 로그가 표시된다. 그러나 Account가 업데이트되지 않는다. → 이는, Hierarchical Injector 와 연관이 있다.

  1. Hierarchical Injector 이해하기

  • accAcount 이벤트가 바인딩되어도 상태변화가 없다. → 코드가 동작하는데 이슈가 있다고 생각할 수 있으나, 이는 정상적으로 동작하는 예이다.

  • Hierarchical Injector의 종류

    • AppModule: 애플리케이션 전체에서 동일한 서비스 인스턴스를 사용할 수 있다.(Singleton Pattern)
    • AppComponent: 모든 컴포넌트에 대해 동일한 서비스 인스턴스를 사용할 수 있다(다른 서비스에서는 동일한 서비스 인스턴스를 사용할 수 없다)
    • Any Other Component: 컴포넌트 및 그 모든 하위 컴포넌트에 대해 동일한 서비스 인스턴스를 사용할 수 있다. (Angular Dependency Injector는 실제로는 Hierarchical Injector이기 때문에)

  • 인스턴스는 위로 전파되지 않고, 해당 컴포넌트의 트리로만 전파되기 때문에 가장 낮은 레벨의 하위 컴포넌트에서 서비스를 주입시킬 경우, 이 컴포넌트 안에서만 사용이 가능하다. 상위 컴포넌트에서 Intector를 했다고 하더라도, 이를 덮어쓴다. (JavaScript Scope)

  • newAccountComponent 이벤트 이슈 처리 방법

    • 현재 newAccountComponent 에서 또한 accountService를 injector를 시키고 있다. AppComponent에서 injector를 하기 때문에 정상이라고 생각할 수도 있지만, newAccountComponent에서 injector된 새로운 인스턴스에 덮혀씌인다.
    • 즉, newAccountComponent의 provider 배열에서 accountService를 제거한다. 그러나 constructor에서는 제거하면 안된다.
  1. Service 안에 Service 주입시키기

  • 서비스 안에 서비스를 삽인하는 건, 컴포넌트 레벨로 제공할 수 없기 때문에 App Module에 서비스를 제공할 필요가 있다. → app.module.ts에 provider 등록하기

  • 서비스 안에서 서비스 호출하는 코드 예시

        @Injectable
        export class AccountsServcie {
          constructor(
            private loggingService: LoggingService
          ) {
          }
    

          addAccount() {
            //...
            this.loggingServcie.logStatusChange(status);
          }
    

          updateStatus() {
            this.loggingServcie.logStatusChange(status);
          }
        }

  • 여기서 중요한 것은, @Injectable 데코레이터를 추가해야한다는 점이다. 이는, Angular에게 이 서비스를 이제 주입할 수 있게 되었거나, 정확히는 이 서비스 안에 무언가를 주입할 수 있게 되었다고 말하는 것이다.

  • 즉, 주입하고 싶은 서비스(주입을 하는 서비스-loggingService가 아님)에 @Injectable를 추가해야한다.

  • 주입이 필요한 서비스에만 @Injectable을 추가하면되지, 주입이 필요없는 서비스에도 @Injectable를 추가할 필요가 없다. 기술적으로 차이는 없지만, 코드를 읽는 사람들에게 알려주기 위함이다.

  1. 서비스의 Cross-Component Communication 사용

  • 서비스를 사용하게 되면, 코드가 더 깔끔하고 중앙 집중적이며 유지보수에 용이하다. 이는, 컴포넌트 A에서 컴포넌트 B로 데이터를 가져오기 위해 이벤트와 속성을 전달하는 복잡한 출력 입력 체인을 구축할 필요가 없다는 뜻이다.

  • Cross-Component Communication 예시

    • 현재 코드 예시: 컴포넌트 A에서 버튼을 클릭 > 이벤트 에미터 > acc로 값을 셋팅 > app-account 컴포넌트로 값을 보냄
        // app.compoennt.html
        <app-account *ngFor="let acc of accounts; let i =index"
                     [account]="acc"
                     [id]="id">
        </app-account>

  • 하나의 컴포넌트에서 트리거하고, 다른 컴포넌트에서 들을 수 있는 이벤트를 제공하고 싶다. → 서비스 내부에서 EventEmitter를 사용

        // account service
        statusUpdate = new EventEmitter<string>();

  • 이벤트 송신 방법

        // account-component.ts
      export class AccountComponent {
        onSetTo(status: string) {
          this.accountsService.updateStatus(this.id, status);
          this.accountsService.statusUpdate.emit(status);
        }
      }

  • 여기서 emit 메서드를 이용하여, 모든 이벤트를 송신한다. 서비스에서도 업데이트가 된다.

  • 이벤트 수신 방법

      export class NewAccountComponent {
        // new-account-component.ts
        contructor() {
          this.accountsServcie.statusuUpdated.subscribe(
            (status: string ) => alert('New Status: ' + status)
          );
        }
      }

  • event emitter는 observable이다. 그래서 status가 업데이트되면, 여기서 새로운 데이터 값을 수신할 수 있다.

  • 이제, 서비스를 이용하여 컴포넌트 간 통신을 할 수 있다. 여기서 중요한 것은, 적절한 서비스의 인스턴스를 생성하고, 서비스를 서비스에 주입하는 경우에는 반드시 앱 모듈 레벨로 서비스를 제공하고, 서비스를 삽입하는 서비스에 @Injectable을 추가해야한다.
