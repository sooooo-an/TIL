5-2. Routing2

  1. Query Parameter & Fragments 전달하기

Query Parameter는 무엇일까? URL에 있는 path 중에 맨 뒤에 ?(물음표)부터 시작되어, key=value 형식을 띄고 있는 부분들을 Query Parameter라고 한다. 그렇다면, 우리는 Angular links을 이용하여 Query Parameter를 어떻게 전달할 수 있을까?

  • /servers/5/edit?allowEdit=1 Navigating 하기
    <a [routerLink]="['/servers', 5, 'edit', '?=']"
       [queryParams]="{allowEdit: '1'}"></a>

  • routerLink 디렉티브에 Query Parameter를 추가하지 못한다. queryParams라는 새로운 디렉티브를 이용하여 Query Parameter를 추가할 수 있다.

  • /servers/5/edit?allowEdit=1#loading Navigating하기

        <a [routerLink]="['/servers', 5, 'edit', '?=']"
           [queryParams]="{allowEdit: '1'}"
           fragment="loading"></a>

  • routerLink 디렉티브에 fragment property를 추가하지 못한다. fragment 디렉티브를 사용할 수 있다.

    • fragment property(#)를 사용하면 특정한 장소로 점프할 수 있다.

  • Method를 이용하여 Navigating 하는 방법

        export class ServerComponent {
          onLoadServer(id: number) {
            this.router.navigate(['/servers', id, 'edit'], {
              queryParams: {allowEdit: '1'},
              fragment: 'loading'
            });
          }
        }

  • Query Parameter & Fragments 구독하는 방법

       export class EditServerComponent implements OnInit {
         ngOnInit() {
           this.route.queryParams.subsucribe();
           this.route.fragment.subsucribe();
         }
       }

  • snapshot을 이용하는 방법 (this.router.snapshot.queryParams, this.router.snapshot.fragment)

  • subscribe를 이용하는 방법

    • 여기에서도 마찬가지로 unsubscribe를 할 필요가 없다. Angular 내장 Observable의 경우, Angular에서 unsubscribe를 해주기 때문이다.

  1. Child(Nested) Routes 셋팅하기

      const appRouters: Routes = [
        {path: 'servers', component: ServersComponent, children: [
            {path: ':id', component: ServerComponent},
            {path: ':id/edit', component: EditServerComponent},
        ]}
      ];
      // server.component.html
      <div>
        <router-outlet></router-outlet>
      </div>

  • Server 컴포넌트의 자식컴포넌트의 경우, children 배열 안에 추가시킨다. server.component.html을 router-outlet으로 대체함으로써 Server 컴포넌트 내부에서 Child Routes를 확인할 수 있게 되었다. → 모든 하위 경로는 /server 경로를 기준으로 한다.

  • this.router.navigate([]) 를 이용하여 자식컴포넌트로 Navigating 하는 방법

        export class ServerComponent {
          onEdit() {
            this.router.navigate(['/servers', this.server.id, 'edit']);
          }
        }

  • 단지, edit 라는 경로를 뒤에 붙여주기만 하면되는데, 왜 상대경로를 사용하지 않을까? (왜 전체 경로를 전부 작성해주었을까?)

    • 여기에 상대경로를 작성하게 되면, 우리는 두번째 인자에 relativeTo 키값을 설정해주어야하기 때문에 절대경로로 작성해주었다.

  1. Query Parameter Configure 핸들링하기

    우리가 상대경로를 사용할 때, 두번째 인자에 relativeTo 키값을 추가하는 것처럼, Query Parameter를 추가할 때도 두 번째 인자에 키값을 추가하여 Query Parameter를 추가하여 navigating 할 수 있다.

      export class ServerComponent {
        onEdit() {
            this.router.navigate(['edit'], {
                    relativeTo: this.route,
                    queryParamsHandling: 'merge'
            });
        }
      }
  • queryParamsHandling: 문자열을 value로 한다. 사용자가 이전에 가지고 있던 정보를 잃지 않도록 하는데 매우 유용하다.
    • merge: 이전 query params와 여기에서 추가되는 query params가 병합되어 사용되어진다.
    • preserve: 기본 동작을 덮어쓰고 이전 동작을 유지하도록 한다. (overwrite the default behavior)
      • 새로운 query params가 추가될 경우, 오래된 query params는 새로운 query params를 덮어쓴다.
      • 새로운 query params가 추가되지 않을 경우, 이전 동작을 유지한다.

  1. Redirecting & Wildcard Routes

    유저가 URL을 임의로 작성하였을 때, 우리가 설정해놓지 않은 Routes일 경우 404 에러페이지로 넘어가도록 구현하고 싶다. 어떻게 할 수 있을까?

      // app.module.ts
      const appRoutes: Routes = [
        { path: 'something', redirectTo: '/not-found' }
      ];

  • 기존의 방식대로 사용한다고 하면, 모든 path에 대해서 어떤 URL을 리다렉트할 건지에 대해서 전부 작성해주어야 하는데, 이는 사실상 불가능하다.

        // app.module.ts
        const appRoutes: Routes = [
          { path: '**', redirectTo: '/not-found' }
        ];

  • **라는 Wildcard Routes를 사용하여 해결할 수 있다.

    • **는 우리가 모르는 모든 경로를 잡는 것을 의미한다. 그렇기 때문에 배열의 순서가 중요한데, 일반적으로는 Routes 배열의 마지막에 작성을 해주어야 한다. 왜냐하면, 알고 있는 경로의 경우일 경우에는 앞 인덱스에서 처리가 되고, 모르는 경로일 경우에는 마지막의 순서로 redirect를 할 수 있는 것이다.
      // app.module.ts
      const appRoutes: Routes = [
        { path: '', redirectTo: '/somewhere-else', pathMatch: 'full' }
      ];

  • 위의 Routes가 존재한다면, Angular는 항상 ‘/somewhere-else’ 로 redirect를 보낼 것이다.

    • 기본 Routes의 일치 전략은 “prefix”이므로 Angular는 URL에 입력한 Route가 지정된 Route로 시작되는지 확인한다 (모든 Routes는 “”로 시작된다)

  • pathMath라는 프로퍼티를 이용하여 해결할 수 있다.

  1. Route Configure 분리하기

    두 개 또는 세 개 이상의 Routes를 가진 경우, app.module.ts에 Routes를 추가하는 것보다 app-routing.module.ts 파일을 생성하여 Routes 관련 모듈만을 작성하는 것이 일반적이다.

      //app-routing.module.ts
      const appRoutes: Routes = [];
    

      @NgModule({
        imports: [
            RouterModule.forRoot(appRoutes)
        ],
        exports: [RouterModule],
      })
      export class AppRoutingModule { }
      // app.module.ts
      @NgModule({
        imports: [AppRoutingModule],
      })
      export class AppModule { }

  2. Guards에 대해서

    현재 각각 컴포넌트에서 path를 체크하고 있는데, 이를 Guards 사용하여 컴포넌트가 로드되기 전에 path를 체크하도록 수정해보자.

  3. Guard - canActivate

    path 중 일부를 보호하기 위해서 사용할 수 있는 Angular 내장 기능으로, Guard를 사용하면 우리가 정의한 시점에 어떤 코드를 실행할 수 있다.

      export class AuthGuard implements CanActivate {
        canActivate(
            route: ActivatedRouteSnapshot,
            state: RouterStateSnapshot
        ): Observable<boolean> | Promise<boolean> | boolean {
        }
      }

  • 위의 route, state 매개변수는 Angular로부터 얻을 수 있는 데이터로, 이는 Routes가 로드되기 전에 얻을 수 있다. 우리는 이를 가지고, boolean 값을 리턴할 수도 있고, 비동기 또는 Observable을 리턴할 수도 있다.

  • canActivate: 비동기로 동작하며, Observable이나 Promise 또는 동기식을 반환한다.

    • 완전하게 클라이언트에서 실행되는 코드를 실행시키는 Guard가 있을 수 있기 때문에 동기적으로 실행되거나 완료하는 데 몇 초가 걸리는 코드가 있을 수 있다.
      • 왜냐하면 timeout을 사용하여 서버에 도달하기 때문에, 비동기로 실행되며 둘 다 canActivate를 통해 가능하다.

  • Auth Guard Example

        //auth.service.ts
        export class AuthService {
          loggedIn = false;
    

          isAuthenicated() {
            const promise = new Promise((resolve, reject) => {
              setTimeout(() => {
                resolve(this.loggedIn);
              }, 800);
            });
          }
        }
        // auth-guard.service.ts
        export class AuthGuard implements CanActivate {
          canActivate(
            route: ActivatedRouteSnapshot,
            state: RouterStateSnapshot
          ): Observable<boolean> | Promise<boolean> | boolean {
            this.authService.isAuthenticated().then((authenticated: boolean) => {
              if (authenticated) {
                return true;
              } else {
                return false;
              }
            });
          }
        }

  • promise를 리턴하여 true를 리턴하여 navigate가 되거나 false를 리턴하여 navigate를 취소시킨다. → 이전의 navigation을 취소시킨다.

  • Route에 Guard 활성화시키는 방법

          //app.module.ts
          const appRouters = [
            {path: 'blrabala', canActivate: [AuthGuard], }
          ];

  • Guard는 자동으로 자식 Routes에 적용된다.

  • canActivateChild 메서드를 이용하여 자식 Routes에만 Guard 설정하기

       export class AuthGuard implements CanActivate, CanActivatedChild {
         canActivate(
           route: ActivatedRouteSnapshot,
           state: RouterStateSnapshot
         ): Observable<boolean> | Promise<boolean> | boolean {
           this.authService.isAuthenticated().then((authenticated: boolean) => {
             if (authenticated) {
               return true;
             } else {
               return false;
             }
           });
         }
    

         canActivateChild(
           route: ActivatedRouteSnapshot,
           state: RouterStateSnapshot
         ): Observable<boolean> | Promise<boolean> | boolean {
           return this.canActivate(route, state);
         }
       }
       //app.module.ts
       const appRouters = [
         {path: 'servers', canActivateChild: [AuthGuard], }
       ];

  • canActivateChild를 추가하면 어떤 이점이 있을까?

    1. CanActivateChild 인터페이스가 추가되었기 때문에(implement CanActivateChild) canActivate 뿐만 아니라 canActivateChild 사용이 가능해졌다.
    2. canActivate를 사용하여 모든 자식 Route에 대한 인증도 가능고, canActivateChild를 사용하여 단일 자식 Route에 대한 인증도 가능해졌다.
    3. 앱에서 필요한 동작에 따라 전체 경로 및 자식 경로를 보호하기 위해 세밀하게 구현가능.

  1. Guard - canDeactivate

    canActivate는 Route 접근 여부에 대한 제어에 초점을 맞추고 있다면, canDeactivate는 Route 이탈 여부에 대한 제어에 초점을 맞추고 있다. input의 값과 같은 사용자 커스텀 값이 변경이 되었을 때, 유저가 뒤로가기 버튼을 클릭하였을 때, 이 값을 업데이트할 건지 안할건지에 대한 모달을 띄우는 등의 방식에 canDeactivate를 사용하면, 사용자가 실수로 다른 곳으로 이동하는 것을 편리하게 방지할 수 있다.

      //can-deactivate-guard.service.ts
      export interface CanComponentDeactivate {
        canDeactivate: () => Observable<boolean> | Promise<boolean> | boolean;
      }
    

      export class CanDeactivateGuard implements CanDeactivate<CanComponentDeactivate> {
        canDeactivate(
            component: CanComponentDeactivate,
            currentRoute: ActivatedRouteSnapshot,
            currentState: RouterStateSnapshot,
            nextState: RouterStateSnapshot
        ) {
            return component.canDeactivate();
        }
      }

  • CanDeactivate는 Angular에서 제공해주는 interface로, generic 타입이다.

    • 어떤 컴포넌트나 클래스가 canDeactivate 메서드를 구현하도록 강제하는 인터페이스를 감싼다. generic 타입을 설정함으로써 나중에 컴포넌트를 여기에서 CanDeactivateGuard에 쉽게 연결할 수 있도록 도와준다.

  • CanDeactivate는 Routes를 떠나려고하면, Angular Router에 의해 호출되는 메서드이다.

    • component: CanComponentDeactivate 타입을 갖는다. 즉, 인자로 들어오는 컴포넌트는, CanComponentDeactivate 인터페이스로 구현된 컴포넌트여야한다.
      • 컴포넌트의 인터페이스와 여기에 작성되어 있는 타입이 동일해야하는 이유: Angular Router가 서비스에서 canDeactivate를 실행할 수 있고, 현재 사용 중이 컴포넌트가 canDeactivate 메서드를 가지고 있다는 점에 의존하고 있기 때문이다. 왜냐하면, Guard와 component 사이에 이러한 연결이 있어야지 Route를 떠날 수 있는지 없는지에 대한 논리를 실제로 구현할 수 있다. (현재 사용중인 컴포넌트와 guard의 연결고리)
    • nextState: Route를 떠나기 전 마지막에 호출된다. (optional)
  //app.module.ts
  const appRouters = [{
    path: 'servers',
    canActivateChild: [AuthGuard],
    children: [
      {path: ':id/edit', canDeactivate: [CanDeactivateGuard], }
    ]
  }];
  // edit-server component
  export class EditServerComponent implements CanDeactivate<CanComponentDeactivate> {
    canDeactivate() {
        if (!this.allowEdit) {
            return true;
        }


        if ((
            this.serverName !== this.server.name ||
            this.serverStatus !== this.server.status
            ) && !this.changeSaved
        ) {
            return confirm('Do you want to discard the changes?');
        } else {
            return true;
        }
    }
  }

  1. Route로 Static Data 전달하기

    Routes 중 일부는 로드될 때마다 정적으로 수신하거나 동적으로 resolve 되는 데이터에 따라 달라진다. 이는 재사용이 가능한 에러페이지를 만드는데 유용하다.

      <!-- error-page.component.html -->
      <h4>{{ errorMessage }}</h4>
      // app.module.ts
      const appRoutes: Routes = [
        {
            path: 'not-found',
            component: ErrorPageComponent,
            data: {message: 'Page not found!'}
        }
      ];

  • 각 Route에 따라 에러메시지를 다르게 보내 에러페이지를 재사용할 수 있다.

      export class ErrorPageComponent implements OnInit {
        ngOnInit() {
            this.errorMessage = this.route.snapshot.data['message'];
            this.route.data.subscribe((data: Data) => {
                this.errorMessage = data;
            });
        }
      }

  1. Guard - resolve Guard

    > 그렇다면 route가 렌더링 또는 화면에 표시되기 전에 가져와야 하는 동적 데이터가 있다면 어떻게 구현할 수 있을까? 예를 들어 클릭하여 로그인하는 동작을 구현하려고 할 때, 버튼을 클릭하면 백엔드에서 개별 서버를 로드하려고 한다. 이 작업은 비동기이기 때문에 몇 초가 걸리는지에 대해 아무도 모른다.
    **이러한 케이스는 `resolver`를 사용하여 해결할 수 있는데,  `resolver` 또한 Route가 렌더링되기 전에 일부 코드를 실행할 수 있는 `canActivate`, `canDeactivate`와 동일하게 서비스라고 생각하면된다.**
  • canActivate vs resolver
    • resolver가 route를 렌더링해야 하는지에 대한 여부, 컴포넌트를를 로드해야 하는지에 대한 여부를 결정하지 않는다. resolver는 항상 컴포넌트를 렌더링하지만 렌더링이 완료되기 마지막에 사전 로딩을 수행한다고 보면 된다. 즉, 컴포넌트의 OnInit 메서드에서 데이터를 가지고 오는동안 spinner를 표시하는 등의 작업을 하는데 유용하다고 볼 수 있다.
    • 결론: canActivate는 컴포넌트를 로드하기 전에 체크하는 것, resolver는 컴포넌트가 초기화될 때 하는 작업.
  //server-resolver.service.ts
  interface Server {
    id: number;
    name: string;
    status: string;
  }


  @Injectable()
  export class ServerResolver implements Resolve<Server>{
    constructor(
        private serverService: ServerService
    ) {}


    resolve(
        route: ActivatedRouteSnapshot,
        state: RouterStateSnapshot
    ): Observable<Server> | Promise<Server> | Server {
        return this.serverService.getServer(+route.params['id']);
    }
  }
  //server.service.ts


  export class ServerService {
    getServer(id: number) {
        const server = this.servers.find((s) => s.id === id);
    }
  }
- `params`가 변경될 때마다 그 값을 구독해서, `serverService`의 `getServer` 메서드로 id를 보내고 있기 때문에 동적으로 동작한다. 이 서비스를 경로를 렌더링할 때마다 실제로 실행되므로 `snapshot`만 있으면 된다. → **컴포넌트 자체와 달리 resolver는 매번 실행되므로 Observable을 사용할 필요가 없다.**
  // server.component.ts


  @Component({
    selector: 'app-server',
    templateUrl: './app-server.component.html',
    providers: [ServerResolver]
  })
  export class ServerComponent {    }
  // app-routing.module.ts


  const appRoutes = [
    {
        path: 'server',
        component: ServerComponent,
        children: [
            {path: ':id', resolve: {server: ServerResolver}}
        ],
    }
  ];
- Router가 로드되면 Angular에 의해 이 메서드(resolver)가 호출된다. resolver의 리턴값은 이 server object에 저장될 것이다. 그리고 우리는 이를 컴포넌트에서 사용할 수 있게 된다.
  //server.component.ts


  @Component({
    selector: 'app-server',
    templateUrl: './app-server.component.html',
    providers: [ServerResolver]
  })
  export class ServerComponent implements OnInit {
    ngOnInit() {
        this.route.data.subscribe((data: Data) => {
                this.server = data['server'];
        });
    }
  }
- 이전과 동일하나 지금은 `server.component`에서 매개 변수를 사용하는 것이 아니라, `resolver`를 사용한다는 점에서 차이점이 있다. (비동기 데이터 사용할 때 중요)
  1. Location Strategies 이해하기

  • 로컬 작업환경에서는 잘 작동하지만, 당연히 여겨서는 안된다. 이와 같은 router, domain/server가 있고, 웹의 실제 서버에서 호스팅하는 경우, 이러한 route들이 작동하지 않을 수도 있다. route, url은 항상 웹이 작동하는 방식이기 때문이다.

  • Angular SPA을 호스팅하는 서버는 404 오류의 경우 또한 설정되어야 한다. index.html 파일을 반환하기 때문에 Angular 앱을 포함하는 파일이 반환된다. 모든 URL의 경우 클라이언트에서가 아닌 서버에서 먼저 구문 분석이 되므로, ‘/server’를 작성하면 웹 앱을 호스팅하는 실제 서버에서 ‘/server’ route를 찾게된다. 그러나 서버에는 해당 route가 없다. Angular 앱이 포함된 html 파일은 index.html 파일 하나 뿐이고, Angular가 이 경로를 인계하고 구문 분석하기를 원하기 때문이다. 그렇기 때문에 호스팅하는 404 오류 페이지에도 index.html으로 경로를 설정해놔야 Angular에게 이 작업을 인계할 수 있게 된다.

  • 그러나 어떤 이유로 인해 이 기능이 작동하지 않거나, 클라이언트에서 이러한 경로를 구문 분석할 수 없는 매우 오래된 브라우저를 지원해야하는 경우(#사용)에는, useHash 프로퍼티를 사용하여 해결할 수 있다.

        //app-routing.module.ts
        @NgModule({
          imports: [Router.Module.forRoot(appRoutes, {useHash: true})],
        })

  • useHash를 추가하게 되면, URL은 URL/#/ 으로 변경되어 있다.

    • #: 웹서버에게 이 hashtag 이전의 URL 부분에만 관심있다는 것을 알려주는 역할을 한다. hashtag 이후의 모든 부분은 웹서버에 의해 무시된다. 따라서 404 오류 발생 시 index.html 파일을 반환하지 않는 서버에서도, hashtag 앞에 있는 URL에만 신경쓰기 때문에 실행시킬 수 있다. 기본적으로 이렇게 작동하며 hashtag 뒤에 부분은 이제 클라이언트, Angular에 의해 구문 분석할 수 있다.
