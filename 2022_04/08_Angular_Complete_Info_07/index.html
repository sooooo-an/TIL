08 Angular Complete Info 07

Observable

1. Observable 이란?

Observable은 기본적으로 data source라고 생각하면 된다. Angular는 기본적으로 Rxjs third-party package를 내장하고 있는데, RxJS에서 Observable은 Observable Pateern으로 구현되어 있다.

  • Observable Pattern

    • Observable과 Observer이 존재한다.

  • observable과 observer 사이에 하나의 타임라인(스트림)이 있다고 생각하고, 이 타임라인에서는 observable 데이터 소스에 따라 observable 데이터 패키지가 방출되는 여러 이벤트를 발생시킬 수 있다.

  • observer: 함수를 구독하는 것과 관련있다.

    • observer가 데이터 패키지를 처리하는 3가지 방법: Handle Data, Handle Error, Handle Completion → 데이터, 오류 수신, 관찰 완료 후에 대한 동작을 결정할 수 있다.

  • 주의사항: 버튼 클릭 이벤트 observable의 경우, 언제 유저가 버튼을 누를지 모르기 때문에 complete를 해서는 안된다. 그러나 http request observable의 경우, 명확한 끝을 가지며, 응답 후에 어떤 동작이 일어난 뒤에 완료되기 때문에 complete를 사용해도 된다.

  • observable 패턴 동작: 비동기 작업(이벤트처리, http 등)을 처리하기 위해 사용된다. 비동기 작업이기 때문에 시간이 얼마나 걸릴지도 모른다. 그렇기 때문에 이러한 이벤트를 기다리지 않는다.

    • 이러한 비동기 작업을 처리하는 방법으로는, callback 이나 promise를 사용했을 수도 있으나 이 대안으로 또한 observable 패턴을 사용할 수도 있다.

    • Observable Pattern의 장점

      • observable은 데이트의 흐름이며, 새로운 데이터가 방출될 때마다 구독하고 있는 observer는 이를 알게된다.
      • 이 스트림은 새로운 페이지로 이동할 때마다 새로운 path의 매개변수를 제공한다. 즉, URL이 변경될 때마다 새로운 데이터가 전달된다. → 아래코드 참고
          export class ServerComponent implements OnInit {
            ngOnInit() {
             this.route.params.subscribe((data) => {
                 console.log(data);
              });
           }
         }

2. Observable 원리 파악하기

 export class ServerComponent implements OnInit {
    ngOnInit() {
      interval(1000).subscribe(count => {
      console.log(count);
    });
   }
 }
  • 다른 컴포넌트로 이동한다고 해서, observable이 value를 방출하는 것을 멈추지 않는다. (그러나 HTTP같은 observable은 done을 한다)
  • 다시 홈페이지에 진입하게 되면, 새로운 observable이 시간된다. 즉 이 말은, 우리는 계속 이전의 observable을 사용하면서 새로운 것 또한 생성시킨다는 건데 이는 백그라운드에서 발생하면 리소스가 빠르게 고갈되고 앱 실행 속도가 느려지는 이슈(불필요한 데이터로 인해 메모리가 많이 소모하기 때문에)가 있다. 그렇기 때문에 컴포넌트가 생명주기가 끝날 때, observable을 구독해지하는 작업이 필요하다. (unsubscribe)

3. Custom Observable 만들기

const customIntervalObservable = Observable.create((observer) => {
  let count = 0;
  setInterval(() => {
    observer.next(count);
    count++;
  }, 1000);
});


customIntervalObservable.subscribe(() => {
  console.log(data);
});
  • observer는 새로운 데이터(next), 오류(error), 완료(complete)에 대한 정보를 얻는데 관심이 있다. 여기서 우리의 일은 observable에게 새로운 데이터, 오류, 완료에 대해서 말하는 것 뿐이다.
  • next: 새로운 데이터를 내보내기 위해 next를 호출한다.
  • error: 오류를 던질 때 사용한다.
  • complete: observable에게 완료되었음을 알릴 때 사용한다.

4. Error & Completion

첫 번째 인자로 들어오는 데이터를 필요로 하는 경우는 90%정도이다. 나머지 10%는 HTTP 요청이나 이에 대한 오류 핸들링에 대해 필요로 하는 경우이다.

  const customIntervalObservable= Observable.create((observer) => {
  let count = 0;
  setInterval(() => {
    observer.next(count);
    if (count === 2) { observer.complete(); }
    if (count > 3) {
      observer.error(new Error('Count is greater 3!'));
    }
  }, 1000);
});


  customIntervalObservable.subscribe((data) => {
    console.log(data);
  }, (error) => {
    console.error(error);
  }, () => {
    console.log('Completed!');
  });
  • error로 방출되는 값은 4이다. observable이 오류를 발생시킬 때마다 observable은 취소 및 완료된다. 따라서 이 경우에는 구독을 취소할 필요가 없다. 위의 방식처럼 observer.error(new Error('Count is greater 3!')); 을 이용하여 오류를 발생시킬 수 있으며, 단순히 오류를 발생시키는 것이 아니라 HTTP 요청과 같이 실패할 수 있는 기본 Angular 기능을 사용할 때 함께 오류를 발생시킬 수 있다.
  • Observable의 두번째 인자: error를 받을 수 있다. → 이 기능을 이용하여 오류 메시지를 표시하고 사용자에게 경고를 표시할 수 있다.
  • Observable의 세번째 인자: Observable을 완료시킬 수 있다.
    • Observable을 완료시킨다는 것은, 오류가 발생하면 구독이 취소되는 것과는 달리, 정상적인 동작이다. 예를 들면, HTTP 요청이 있는데, HTTP 요청의 경우 데이터를 받으면 완료가 된다. 즉, 서버에 의한 응답이 있을 때마다 완료가 된다.
    • 즉, 우리는 Observable의 행동을 정의하고, 구독하며 오류가 발생시키기 전에 Observable을 완료시킬 수 있다는 것이다.
    • Observable이 완료가 된 다음에 발생되어야 하는 동작을 3번째 인자에 추가할 수 있다. → 동작을 초기화하거나 완료 후 동작되어야 할 일(완료가 된 Observable의 경우, unsubscribe를 할 필요가 없다.)
    • 그러나 오류가 발생하여 완료가 되었을 경우, 3번째 메서드에 작성되어 있는 동작은 확인할 수 없다. 즉, 오류는 구독을 취소하지만 완료시키지는 못한다.

5. Operator 이해하기

Observable 안에서 어떤 일이 일어나는 지 이해해야한다. 구독하고 다른 핸들러 기능을 설정할 때마다 RxJS는 결국 그것들을 모두 하나의 객체로 병합하고 그 객체인 observer를 Observable 그리고 Observable 내부로 전달한다.

  • RxJS의 라이브러리를 사용하여, 우리가 사용하고 싶은 데이터구조로 변경하여 사용할 수 있다. (map, filter 등)
  • pipe method: 모든 Observable은 pipe 메서드를 내장하고 있다.

6. Subject 이해하기

서비스에서 EventEmitter하는 방식을 배웠다. 그러나 EventEmitter를 사용하는 것보다 Subject를 사용하는 것이 더 좋다.

  • Observable은, 콜백이나 이벤트를 하나 더 감싸는 것이다. Subject는 다 다르고, Subject는 구독할 수 있는 객체이다. 그러나 외부에서 호출할 수 있기 때문에 EventEmitter보다 더 작업하기가 수월하다.
  • Observable에서 next를 호출했을 때도, next는 Observable 내부에서 호출할 수 있다. 그렇기 때문에 Subject는 EvenEmitter를 사용이 필요한 상황에서 보다 완벽하게 사용할 수 있다. 즉, HTTP 요청이나 DOM 이벤트와 같은 이벤트 소스가 아니라 능동적으로 처리해야하는 데이터일 경우, Subject를 사용하면 좋다.
  • EventEmitter를 사용하지 말고, Subject를 사용해라
    • Observable이 사용할 수 있는 Operator를 전부 사용할 수 있다.
    • Observable과 마찬가지로 Subject가 필요하지 않을 때마다 Observable을 unsubscribe해야한다.
