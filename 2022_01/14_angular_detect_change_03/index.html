Angular Detect Changes 3

1. OnPush change detection 모드

Todo 목록이 너무 커진 경우 Todo 목록이 변경될 때만 TodoList 컴포넌트가 자체 업데이트되도록 설정할 수 있다. 이 작업은 컴포넌트 change detection 전략을 OnPush로 업데이트하여 수행할 수 있다.

  • Todo 코드예시
@Component({
    selector: 'todo-list',
    changeDetection: ChangeDetectionStrategy.OnPush,
})
export class TodoList {}

우리의 애플리케이션에 2개의 버튼을 추가해보자. 하나는 직접 변경하여 목록의 첫 번째 아이템을 토글하는 것이고, 다른 하나의 버튼은 전체 리스트에 Todo를 추가하는 것이다.

  • Todo 코드 + 2개 버튼 추가 코드 예시
@Component({
    selector: 'app',
        template: `<div>
                                        <todo-list [todos]="todos"></todo-list>
                             </div>
                             <button (click)="toggleFirst()">Toggle First Item</button>
                             <button (click)="addTodo()">Add Todo to List</button>
                                `
})
export class App {
    todos: Array = initialData;
    constructor() {
    }


    toggleFirst() {
        this.todos[0].completed = !this.todos[0].complated;
    }


    addTodo() {
        let newTodos = this.todos.slice(0);
        newTodos.push(new Todo(1, "TODO 4", false, new Owner("John", "Doe")));
        this.todos = newTodos;
    }
}
  1. 첫 번째 버튼, “Toggle First Item”은 작동하지 않는다. 그 이유는 toggleFirst() method가 직접적으로 리스트의 엘리먼츠를 변형하기 때문이다.
    • input reference가 변경되지 않았기 때문에 TodoList는 이를 감지할 수 없다.
  2. 두 번째 버튼은 작동한다. addTodo() method는 todo list의 복사본을 만든 다음, 그 아이템을 복사본에 추가하고, 마지막으로 todos의 변수를 복사된 리스트로 바꾼다.
    • 컴포넌트가 input에서 reference 변경을 인지하기 때문에 change detection이 트리거된다. 새 목록을 수신한다.
  • todos list를 직접 변형시킬 수 없다. 새 리스트가 필요하다.

2. OnPush는 정말로 input을 reference를 통해 비교하는 것일까?

그렇지 않다. todo를 클릭하여 토글을 시도할 때도, 계속 작동한다. TodoItem을 OnPush로 전환하는 경우에도 마찬가지이다. 그 이유는, OnPush는 컴포넌트 input의 변경 사항(컴포넌트가 change detection를 트리거하는 이벤트를 내보내는 경우)만 확인하지 않기 때문이다.

  • Victor Savkin이 블로그에서 인용한 내용 OnPush detectors를 사용할 때, 프레임워크는 input의 프로퍼티가 변경되거나 이벤트가 발생할 때, 또는 Observable이 이벤트를 실행할 때, OnPush 컴포넌트를 체크한다.

OnPush를 사용하면 성능이 향상되지만, mutable object와 함께 사용할 경우, 복잡성 비용이 많이 들게 된다. 그것은 추론하고 번식하기 어려운 버그를 만들게 된다. 하지만 Immutable.js를 사용하게된다면 이를 방지할 수 있다.

3. Angular 앱의 구축을 단순화하기 위한 Immutable.js 사용

immutable object만 사용하여 애플리케이션을 구축하면, change detection 버그에 걸려들 위험 없이 어떤 컴포넌트에서나 OnPush를 사용할 수 있게되는데, 그 이유는, 이는 immutable object의 경우, data를 수정하는 유일한 방법이 새로운 객체를 생성하고, 이전 객체를 대체하는 것이기 때문이다.

새로운 immutable 객체는 항상 OnPush change detection을 트리거한다. 우리는 객체의 새로운 복사본을 만드는 것을 하지 않음으로써 우연히 버그를 만들 수 없다. 왜냐하면, 데이터를 수정하는 유일한 방법은 새로운 객체를 만드는 것이기 때문이다.

좋은 선택은 immutable.js 라이브러리를 사용하는 것이다. 이 라이브러리는 immutable objects(Map) 등의 메소드를 통해 immutable primitives 앱을 만드는 데 도와준다.

이 라이브러리는 type-safe 방법으로도 사용할 수 있다.

4. change detection 루프 방지: 운영 및 개발 모드

Angular Change Detection의 중요한 속성 중 하나는 AngularJS와 다르게 단방향 데이터 흐름을 적용한다는 것이다. 즉, 컨트롤러 클래스의 데이터가 업데이트되면, change detection이 실행되고 업데이트된다.

그러나 뷰를 업데이트한다고 해서 추가 변경이 발생하지는 않으며, 이 변경은 다시 뷰에 대한 추가 업데이트를 트리거하여 AngularJS를 다이제스트 사이클이라고 한다.

5. Angular에서 change detection 루프가 트리거 되는 방법

change detection 루프가 트리거되는 한 가지 방법은 라이프사이클 콜백을 사용하는 것이다. 예를 들어, TodoList 컴포넌트에서 바인딩 중 하나를 변경하여 다른 컴포넌트에 대한 콜백을 트리거할 수 있다. 

ngAfterViewChecked() {
    if (this.callback && this.clicked) {
        console.log("changing status...");
        this.callback(Math.random())
    }
}
  • EXCEPTION: Expression '{{message}} in App@3:20' has changed after it was checked
    • 이 에러 메시지는 Angular를 개발 모드로 실행할 경우에만 발생한다. production 모드를 활성화하면 어떻게 될까?
@NgModule({
    declarations: [App],
    imports: [BrowserModule],
    bootstrap: [App]
})
export class AppModule {}

production 모드에서, 에러는 발생하지 않고, 문제가 감지되지 않은 상태로 유지된다.

6. change detection 이슈가 자주 발생할까?

change detection loop를 트리거하기 위해 정말 노력해야하지만, 개발 단계에서 항상 개발 모드를 사용하는 것이 더 나은 경우를 대비하여 문제를 피할 수 있다.

이 보장은 Angular가 change detection을 항상 두 번 실행하는데, 이 이슈는 2번째 실행하는 change dection 사례이다. production 모드에서 change detection이 한 번만 실행됩니다.

7. change detection 감지 켜기/끄기 및 수동 트리거

change detection을 해제하고 싶은 특별한 경우가 있을 수 있다. 백엔드에서 웹소켓을 통해 많은 데이터가 수신되는 상황을 상상해보자. 5초마다 한 번씩 UI의 특정 부분을 업데이트하고자 할 수도 있다. 이를 위해 먼저 컴포넌트에 change detector를 주입시킨다.

  • change detector 주입 예시
constructor(private ref: ChangeDetectorRef) {
    ref.detact();
    setInterval(() => {
        this.ref.detectChange();
    }, 5000);
}

보시다시피 change detector를 분리하기만 하면, change detector가 꺼진다. 그 다음 detectChange를 호출하여 5초마다 수동으로 트리거한다.

8. 요약

Angular change detection은 컴포넌트의 데이터와 컴포넌트의 HTML 템플릿 뷰 간의 자동 동기화를 보장하는 기본 제공 프레임워크 기능이다.

change detection은 마우스 클릭, HTTP 요청 및 기타 유형의 이벤트와 같은 일반적인 브라우저 이벤트를 탐지하고 각 컴포넌트의 뷰를 업데이트해야 하는지 여부를 결정하는 방식이다.

  • 2가지 유형의 change detection
  • 기본 change detection
    • Angular는 컴포넌트 트리의 모든 컴포넌트에 대해 이벤트 발생 전후의 모든 템플릿 표현식 값을 비교하여 뷰를 업데이트하는 지에 대한 여부를 결정한다.
  • OnPush change detection
    • 일부 새 데이터가 컴포넌트 input 또는 비동기 파이프를 사용하여 구독된 관찰 가능을 통해 컴포넌트에 명시적으로 푸시되었는지 감지하여 작동한다.

Angular 기본 detection change 메커니즘은 실제로 AngularJS와 매우 유사하다. 즉, 브라우저 이벤트 전후의 템플릿 식 값을 비교하여 변경된 사항이 있는지 확인한다. 모든 컴포넌트에 적용된다.

  • AngularJS와 Angular의 detection change 메커니즘의 차이점
  • AngularJS
    • detection change loop나 다이제스트 사이클이 없다. ⇒ 따라서 템플릿과 컨트롤러만 보고 각 컴포넌트를 구성할 수 있다.
    • change detection 메커니즘이 Angular보다 느리다
    • AngularJS는 사용자 정의가 불가능하나, Angular의 change detection 메커니즘은 사용자 정의가 가능하다.

9. change detection에 대해 많이 알아야할까?

아마도 애플리케이션 및 사용 사례의 95%의 경우 Angular Change Detection은 단지 작동하며 우리가 알아야 할 것이 없다고 해도 무방하나, 몇 가지 이유로 인해 작동 방식에 대한 아이디어를 얻는 것은 여전히 유용하다

  • change detection loop와 관련된 것과 같은 개발 오류 메시지를 이해하는 데 도움이 된다.
  • 오류 스택 추적을 읽는데 도움이된다.
  • 성능이 우수한 경우, change detection에 대해 알고 있으면 성능 최적화를 하는데 도움이 된다
