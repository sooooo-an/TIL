02 Rxjs Study Handling Multiple Pipe

RxJS Study (zip, combineLatest, withLatestFrom, forkJoin)

통합알림개편 프로젝트에서 추후에 추가된 요구사항으로는 밑에 있는 사항이다.

💡 "액티비티 리스트 클릭 시, 유효성 검증에 실패한 리스트의 경우 에러 모달을 띄어주고, 검증에 실패하지는 않았지만 삭제가 되어 진입이 불가능한 리스트의 경우에는 읽었다는 액션만 주고, 이동을 하지 않는다.

이 요구사항을 해결하기 위해서는 먼저, 2가지 조건을 확인하는 것이 필요하다. 먼저, 클릭 된 액티비티 리스트의 activity_id 의 값을 보내 검증을 해야하고, 두 번째로는, 클릭 된 액티비티의 code 값을 스토어에서 확인하여, move_yn 의 키 값을 확인 해야한다.

  • 클릭 된 액티비티 리스트의 activity_id 의 값을 보내 검증: 액티비티의 리스트는 실시간이 아니기 때문에, 다른 유저가 자신의 글을 삭제하면서 진입이 불가능할 경우에도 클릭 시 진입이 되어 "진입할 수 없습니다" 등의 에러 모달이 뜨게 된다. 결국 유저에게 한 번 더 진입을 하게 됨으로써 유저의 경험을 더 악화시킬 수 있으므로, 클릭 할 때 검증을 통해서 진입이 가능한지 먼저 확인한다.
  • 클릭 된 액티비티의 code 값을 스토어에서 확인하여, move_yn 의 키 값을 확인: 처음 진입 시 서버에서 불러온 static한 code 값을 global store에 저장하게 되는데, 각 code들이 네비게이션을 통해 진입이 가능한 코드인지, 불가능한 코드(삭제된 채널입니다, 삭제된 프로젝트입니다 등)인지까지 저장을 하고 있다. 그렇기 때문에 이 스토어를 구독함으로써 클릭한 액티비티의 move_yn를 가지고 와 이동을 시킬 지, 시키지 않을 지에 대한 선택을 하게 됨으로 유저 또한 쓸데없는 이동을 방지할 수 있게 된다.

이 두 가지의 체크사항을 보면, 두 개의 스트림이 딱히 시간관계로 묶여있지 않음을 볼 수 있다. 클릭을 하였을 때, 동시에 이루어져도 문제가 없음을 알 수 있다. (근데, validation에서 실패하였을 경우에는 굳이 store에서 가지고 올 필요가 없는데 액티비티 리스트 검증을 끝낸 뒤에 스토어에서 가지고 오는 게 더 낫지 않을까?)

그렇기 때문에 RxJS의 combineLatest와 forkJoin 중에 어떤 것을 사용할까 고민을 하다, 내가 이 내용을 정확하게 알고 있지 않다는 결론이 나왔다. zip과 combineLatest, forkJoin, withLatestFrom에 대해서 공부해보자.

블로그와 동일하게 티셔츠의 색상과 로고를 가지고 예시를 만들어보겠다. 추후에 더 좋은 예시를 만들어서 변경해보자.



import { forkJoin, zip, combineLatest, Subject } from 'rxjs';
import { withLatestFrom, take, first } from 'rxjs/operators';


type Color = 'white' | 'green' | 'red' | 'blue';
type Logo = 'fish' | 'dog' | 'bird' | 'cow';


const color$ = new Subject<Color>();
const logo$ = new Subject<Logo>();


// checking //


color$.next('white');
logo$.next('fish');


color$.next('green');
logo$.next('dog');


color$.next('red');
logo$.next('bird');


color$.next('blue');

01. Zip

zip은 스트림이 모두 도착하면, 동작을 하게 되는 오퍼레이터이다. 각 스트림의 값이 모두 변경이 있어야 동작하게 된다.

zip(color$, logo$)
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));


// white shirt with fish
// green shirt with dog
// red shirt with bird
  • color는 logo의 새로운 값이 들어올 때까지 기다린다. color와 logo가 새로운 값으로 둘 다 변경하면, 그 때 log가 동작하게 된다.

02. combineLatest

combineLatest는 독립적인 오퍼레이터이다. 각각의 값을 기다리지 않는다. 스트림의 처음의 값만 서로 만나게 되면, 각각의 스트림이 울릴 때마다 값이 합쳐진채로 동작하게 된다.

combineLatest(color$, logo$)
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));


// white shirt with fish
// green shirt with fish
// green shirt with dog
// red shirt with dog
// red shirt with bird
// blue shirt with bird
  • 첫 번째 함수는 color와 logo 값이 모두 변경될 때 동작하게 된다. 그 이후로는 color나 logo 중 하나의 값만 변경해도 그 때 log가 동작하게 된다.
    1. Ms. Color picks WHITE
    2. Mr. Logo picks FISH ⇒ log 01, color와 logo 값이 처음으로 만났다 (WHITE + FISH)
    3. Ms. Color picks GREEN ⇒ log 02, color의 값만 변경되었지만 이전의 logo값과 조합되어 동작된다. (GREEN + FISH)
    4. Mr. Logo picks DOG ⇒ log 03, 전의 color 값과 조합되어 동작 (GREEN + DOG)
    5. Ms. Color picks RED ⇒ log 04, RED + DOG
    6. Mr. Logo picks BIRD ⇒ log 05, BIRD + RED
    7. Ms. Color picks BLUE ⇒ log 06, BLUE + BIRD

03. withLatestFrom

withLatestFrom은 primary/secondary 오퍼레이터로 구분되어 있다. 처음에 두 스트림이 꼭 만나야하며, 그 후에는 primary의 값이 변경될 때만 트리거된다.

color$.pipe(withLatestFrom(logo$))
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));


// green shirt with fish
// red shirt with dog
// blue shirt with bird
  • color$: primary / logo$: secondary
    • 처음에 secondary가 응답을 하게 되면, primary는 주도하게 됩니다. 로그는 primary 값이 변경될 때 동작을 하게 되는데, secondary 값은 변경이 되어도 동작하지 않습니다.
  • 동작원리
    1. Ms. Color picks WHITE ⇒ 어떤 일도 일어나지 않는다. secondary의 값을 기다릴 뿐!
    2. Mr. Logo picks FISH ⇒ secondary의 값을 찾았다. secondary는 primary의 명령을 기다린다.
    3. Ms. Color picks GREEN ⇒ log 01, primary는 GREEN 을 명령. GREEN + FISH
    4. Mr. Logo picks DOG ⇒ wait secondary
    5. Ms. Color picks RED ⇒ log 02, primary 는 RED를 명령. RED + DOG
    6. Mr. Logo picks BIRD ⇒ wait secondary
    7. Ms. Color picks BLUE ⇒ log 03, primary는 BLUE를 명령. BLUE + BIRD

04. forkJoin

모든 스트림이 완전히 종결되었을 때만 커밋하는 오퍼레이터이다.(최종 목적지 연산자)

forkJoin(color$, logo$)
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));
  • 어떠한 일도 트리거되지 않는다. color와 로고 observable 모두 종결되지 않았기 때문이다.
    • "종결되지 않았다": next 메서드를 호출하여 값을 변경시킬 수 있다는 뜻이다.
  • 그렇다면, forkJoin이 트리거될 수 있도록 하려면 어떻게 해야할까?

forkJoin(color$, logo$)
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));


color$.complete();
logo$.complete();


// blue shirt with bird

  • 동작원리

    1. Ms. Color picks WHITE
    2. Mr. Logo picks FISH
    3. Ms. Color picks GREEN
    4. Mr. Logo picks DOG
    5. Ms. Color picks RED
    6. Mr. Logo picks BIRD
    7. Ms. Color picks BLUE
    8. Ms. Color completed ⇒ color 스트림의 종결
    9. Mr. Logo completed ⇒ log 01, color와 logo 스트림 모두 종결

  • 자동으로 스트림을 종결시켜주는 오퍼레이터를 사용하여 만들어보자

// 자동으로 종결시켜주는 오퍼레이터 => take, takeUntil, first
const firstColor$ = color$.pipe(take(1));
const firstLogo$ = logo$.pipe(first());


forkJoin(firstColor$, firstLogo$)
    .subscribe(([color, logo]) => console.log(`${color} shirt with ${logo}`));

4가지 오퍼레이터 중에, 내가 사용할 수 있는 오퍼레이터는 무엇일까?

액티비티 리스트를 클릭 시, API는 단 한번의 호출만 필요하고, store의 code 또한 클릭된 액티비티 리스트의 code만 필요하다. 즉 단 한 번만 호출하여 사용해도 된다는 것이다. 그러므로, 내가 사용할 가장 최적의 오퍼레이터는 forkJoin 오퍼레이터이다. 한 번씩만 호출한 뒤 나머지 값의 변경은 구독할 필요가 없기 때문이다.
